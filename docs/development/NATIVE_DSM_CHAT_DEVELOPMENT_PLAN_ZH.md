# Synology Chat 原生聊天功能开发计划

> 文档状态：开发中；共同契约、Apple 领域模型、关闭型网络适配器和 macOS“消息”入口已实现，真实协议待 CH0 实机验证
> 首个参考平台：macOS  
> 目标平台：macOS、iPhone、iPad、Android、Windows  
> 最后更新：2026-07-22

## 1. 产品目标

在岚仓内加入一个聚焦的 Synology Chat 原生聊天模块，只实现以下用户功能：

1. 用户与用户之间的一对一聊天。
2. 选择多个用户创建私人群聊。
3. 发送和接收文字消息。
4. 发送、预览、播放和保存图片。
5. 发送、预览、播放和保存视频。
6. 发送、下载和打开普通文件附件。
7. 在文字消息中输入和显示 Unicode 表情。
8. 录制、发送和播放语音消息。
9. 为指定消息设置个人提醒，并在到期时提示用户。
10. 在一对一聊天或群聊中创建投票并参与投票。
11. 使用 Chat Server 支持的加密会话安全收发消息和附件。

本计划中的“群聊”指私人多人会话，不是公开频道。创建群聊时支持选择成员和填写群聊名称；群聊创建后可以查看名称和成员。首版不扩展为频道管理或组织管理功能。

### 1.1 当前实现基线

- 已建立用户、会话、附件、消息分页、提醒、投票、加密状态和逐项能力的共同 JSON Schema。
- Apple 共享层已建立相同领域模型、输入校验和 `ChatRepository` 边界，并以正式自动化测试覆盖群聊、消息、Emoji、投票和未知字段兼容。
- 已实现独立的 `DsmChatRepository`，通过 DSM 登录会话和 `SYNO.API.Info` 动态发现 `SYNO.Chat.Channel`、`SYNO.Chat.Channel.Named`、`SYNO.Chat.User`、`SYNO.Chat.User.Avatar`、`SYNO.Chat.Post` 与 `SYNO.Chat.Post.Reminder`。这些均为未公开内部接口，只在 NAS 明确返回兼容版本时启用。
- 已接入真实用户列表、会话列表、历史分页、现有一对一会话打开、文字/Unicode Emoji 发送、私人群聊创建与邀请、个人消息提醒写入；写操作使用客户端请求 ID 做进程内重复提交保护，群聊创建后复查成员。
- macOS 已实现会话列表、消息列表、新建单聊/群聊、文字输入、附件选择、加载/空内容/错误状态及能力控制，并在登录后注入真实 Chat Repository；实机反馈已确认数据可加载。
- 第二轮实机修正进一步兼容用户根数组、单数/复数容器、对象字典、空显示名及更多用户/发送者字段；用户目录会回填历史消息发送者，已声明头像通过 `SYNO.Chat.User.Avatar.get` 受限读取，每条消息直接显示完整年月日时分秒，空数据时保持全高对齐。上述结果仍需在当前测试 NAS 复验。
- Chat Server 没有发现创建匿名一对一会话的独立方法。当前可打开并发送到已经存在的双人会话；若两人从未聊天，先在群晖 Chat 中发起一次，不能用命名群聊伪造一对一会话。
- 已将 `SYNO.Chat.Channel.Anonymous` 加入只读能力发现。它可能对应首次一对一会话，但写入方法、参数和幂等语义完成实机确认前不会调用。

为完成上述功能，客户端必须具备账号访问、用户列表、消息历史分页、发送状态、失败重试、增量同步和本地缓存等基础能力。这些属于聊天正常工作的技术底座，不视为额外产品功能。

## 2. 明确不做的功能

以下能力不属于当前开发范围，不作为首版发布条件：

- 公开频道、频道加入、频道归档和频道权限管理。
- 消息讨论串、回复层级、转发、编辑、删除、置顶和书签。
- 消息表情回应、动态贴纸、GIF 商店和自定义表情包。
- 语音通话、视频通话和会议。
- 定时消息、全局搜索和在线状态。
- Bot、Webhook、Slash Command 和第三方集成。
- 公开频道和频道加密；当前加密范围只覆盖一对一与私人群聊。
- Chat Server 管理控制台、套件安装和服务端用户管理。
- 系统通知和后台推送；后续如有需要再单独立项。

“表情”在本计划中仅指输入框中的 Unicode Emoji，例如系统表情键盘或原生表情选择器。Emoji 可以出现在消息正文中，但不得用 Emoji 代替导航、附件、发送等界面图标。

## 3. 接口边界

群晖公开的 `SYNO.Chat.External` 主要用于 Incoming Webhook、Outgoing Webhook、Bot 和 Slash Command，并只能读取机器人可见的数据：

<https://kb.synology.com/en-global/DSM/help/Chat/chat_integration?version=7>

它不能满足普通用户之间聊天、创建私人群聊和以当前用户身份发送附件的需求。因此本功能预计需要使用 Chat 套件客户端采用的未公开接口或协议。

实施要求：

- 使用独立的 `ChatClientInternalAdapter`，不与 File Station 或公开 Bot API 混用。
- 内部 API 名称和版本范围集中在网络适配层；实际路径与可用版本必须来自当前 NAS 的能力发现结果，不硬编码套件路径。
- 每项内部能力按 DSM build 和 Chat Server 套件版本进行能力发现、契约测试和功能开关保护。
- 新版本未经验证时关闭发送、建群、提醒、投票和加密写操作，并给用户明确提示；不能静默尝试未知接口。
- Bot Token 不能模拟普通用户，当前范围也不实现 `SYNO.Chat.External` 集成界面。

## 4. 用户流程与界面

### 4.1 信息架构

Chat 是岚仓的一级功能，面向用户显示为“消息”。

```text
消息
  -> 会话列表
    -> 一对一聊天
    -> 私人群聊
      -> 消息历史
      -> 文字输入
      -> 表情输入
      -> 图片 / 视频 / 文件 / 语音消息
      -> 提醒 / 投票
      -> 加密会话解锁
  -> 新建聊天
    -> 选择一个用户
    -> 选择多个用户并创建群聊
```

- macOS 使用 SwiftUI 原生侧栏、会话列表、消息列表、工具栏、文件选择器和媒体播放能力。
- iPhone 使用原生层级导航和系统返回手势。
- iPad 使用自适应分栏，支持键盘、指针和拖放。
- Android 使用 Jetpack Compose 自适应导航、系统文件选择器和预测性返回。
- Windows 使用 WinUI 3 的 NavigationView、列表、命令栏和系统文件选择器。

开始界面实现前，必须基于岚仓现有品牌和平台语义色生成消息模块设计系统，明确间距、字体角色、浅色/深色、发送状态、附件卡片、加载状态和动效。设计系统只用于开发期，不加入 App 运行时依赖；通用设计建议与平台规范冲突时，以 SwiftUI、Jetpack Compose 或 WinUI 的平台原生体验为准。

### 4.2 会话列表

- 每个项目显示用户或群聊名称、头像、最后一条消息摘要和时间。
- 图片、视频和文件使用通俗摘要，如“图片”“视频”“文件”，不显示接口类型。
- 当前会话有明确选中状态，不能只靠颜色区分。
- 列表为空时显示“还没有聊天”，并提供“新建聊天”操作。
- 会话列表和每个会话的滚动位置在模块切换后保持。

### 4.3 新建聊天与群聊

- 用户可以查看当前账号有权联系的 Chat 用户。
- 选择一个用户时进入一对一聊天；已有会话时进入现有历史，不创建重复会话。
- 选择多个用户时显示群聊名称字段和已选成员列表。
- 创建按钮在提交期间禁用并显示进度，超时后先查询结果，避免重复创建群聊。
- 无权限、用户已停用或创建失败时说明发生了什么以及下一步可以怎么做。

### 4.4 聊天内容与发送状态

- 消息按服务器确认的顺序展示，历史记录按需向上分页。
- 自己和其他用户的消息有清晰区分，同时向屏幕阅读器提供发送者信息。
- 发送状态分为“正在发送”“已发送”“发送失败”；失败消息由用户明确点击重试。
- 网络断开时保留尚未发送的文字，不在后台静默重复提交。
- 切换会话后保留每个会话的未发送文字草稿。
- 发送按钮、附件按钮和表情按钮使用平台原生矢量图标及可访问性标签。

### 4.5 图片、视频和文件

图片：

- 通过系统选择器选择，发送前显示缩略图、大小和移除按钮。
- 消息内按可见范围加载缩略图，点击后使用原生查看器预览。
- 支持保存到用户选择的位置；不得静默写入任意目录。

视频：

- 发送前显示封面、时长、大小和移除按钮。
- 消息内显示封面和播放状态，点击后使用平台原生播放器。
- 视频加载支持取消，离开播放器后及时释放资源。

普通文件：

- 显示文件名、类型、大小、传输进度和失败状态。
- 下载前由用户选择位置或遵循平台明确的下载设置。
- 下载完成后允许打开文件或显示所在位置。
- 文件名、保存路径和响应类型必须校验，避免路径穿越和意外覆盖。

### 4.6 表情

- 优先支持系统表情键盘；桌面端提供系统原生的表情选择入口。
- 消息模型和存储统一使用 UTF-8，正确处理组合 Emoji、肤色修饰符、家庭组合和国旗。
- 字符限制必须按用户感知字符处理，不能把一个组合 Emoji 截断成无效序列。
- 表情与普通文字可在同一条消息中发送。

### 4.7 可访问性与主题

- 支持浅色和深色模式，正文对比度至少 `4.5:1`。
- 支持键盘完整操作、VoiceOver/屏幕阅读器、动态文字和降低动态效果。
- iOS/iPadOS 触控区域至少 `44×44 pt`，Android 至少 `48×48 dp`。
- 图片和视频缩略图提供类型、发送者和状态描述；文件卡片读出文件名、大小和传输状态。
- 发送、上传和下载状态变化不得只靠颜色表示。
- 消息列表使用惰性或虚拟化容器，历史载入时保持阅读位置，避免内容跳动。

### 4.8 语音消息

- 首次录音前使用平台权限说明请求麦克风权限；拒绝后说明如何在系统设置中重新开启。
- 录制界面显示时长、暂停或取消状态，并提供明确的“发送”操作，不能停止录音后自动发送。
- 发送前允许试听和重新录制；上传中显示进度并支持取消。
- 消息内显示播放、暂停、时长和播放位置，离开会话后及时停止并释放音频资源。
- 不实现实时语音通话，麦克风只在用户明确开始录音后启用。

### 4.9 消息提醒与投票

提醒：

- 用户可为自己可见的指定消息选择提醒时间。
- 提醒只属于当前用户，不改变其他成员的消息内容或状态。
- 创建、修改和取消都要核对服务器结果；到期提示不得泄露锁屏隐私内容。
- 当前范围不包含定时发送消息。

投票：

- 一对一聊天和私人群聊均可创建投票，是否允许以 Chat Server 实际权限为准。
- 支持问题、至少两个不重复选项、单选或多选，以及服务器支持时的匿名和截止时间。
- 创建前显示预览；提交期间禁用重复创建；投票后刷新服务器结果。
- 结果不能只靠颜色区分，必须同时显示选项文字和票数或比例。

### 4.10 加密会话

- 只复现 Chat Server 已支持的一对一与私人群聊加密协议，不自行设计不兼容的加密格式。
- 覆盖首次设备、解锁口令、设备加入、私钥恢复、密钥轮换、成员变化和设备撤销。
- 消息正文、图片、视频、文件和语音消息的加密边界必须分别验证。
- 密钥、口令和恢复材料进入平台安全存储，不得写入普通缓存、通知或日志。
- 密钥缺失、损坏或口令错误时显示恢复路径；任何失败都不得回退为明文发送、明文缓存或明文预览。

## 5. 架构设计

```text
原生消息 UI
  -> Chat Presentation / ViewModel
    -> Chat Repository
      -> ChatClientInternalAdapter
        -> Chat Capability / Session / Sync
          -> Attachment Transport
            -> Synology Chat Server
```

| 模块 | 职责 |
| --- | --- |
| Chat Core | 用户、会话、群聊、成员、消息和附件领域模型 |
| Chat Capability | 套件、版本、权限、端点和逐项能力发现 |
| Chat Session | DSM 会话与 Chat 用户会话的交换、续期、退出和清理 |
| Chat Conversation | 一对一会话、私人群聊创建和会话列表 |
| Chat Messaging | 文字与 Emoji 消息、发送状态、幂等和历史分页 |
| Chat Attachment | 图片、视频、文件上传下载、预览、缓存和取消 |
| Chat Voice | 麦克风权限、录音、试听、上传、播放和资源释放 |
| Chat Reminder | 消息提醒创建、修改、取消和到期处理 |
| Chat Poll | 投票创建、参与、结果同步和截止状态 |
| Chat Encryption | 设备密钥、会话密钥、解锁、恢复、轮换和安全存储 |
| Chat Sync | 增量同步、排序、去重、断线恢复和草稿 |

Chat 领域层不依赖平台 UI 和 HTTP 实现。Apple、Android、Windows 共享 JSON Schema、错误语义、同步规则和验收标准，不共享跨平台 UI 运行时。

### 5.1 计划契约

在 `contracts/schemas/` 中逐步建立：

- `chat-capability.schema.json`
- `chat-user.schema.json`
- `chat-conversation.schema.json`
- `chat-group.schema.json`
- `chat-message.schema.json`
- `chat-attachment.schema.json`
- `chat-voice-message.schema.json`
- `chat-reminder.schema.json`
- `chat-poll.schema.json`
- `chat-encryption-state.schema.json`
- `chat-message-page.schema.json`
- `chat-sync-cursor.schema.json`

具体字段必须以完全脱敏的实机响应为准。新增字段默认可选，未知字段必须可忽略。消息、会话、群聊和附件写操作使用客户端请求 ID 做重复提交保护，并在超时后查询服务器最终结果。

## 6. 认证、同步与缓存

- 连接 NAS 后只读检查 Chat Server 是否安装、运行以及当前账号是否有权限。
- 不能假定现有 DSM SID 可直接访问 Chat；先验证 Chat 会话交换、续期和退出行为。
- CH0 先确认真实同步方式，不预设 WebSocket、SSE、长轮询或普通轮询。
- 首次进入只读取当前视窗所需消息，历史内容按需分页。
- 使用服务器游标或经验证的稳定排序键增量同步，以服务器消息 ID 去重。
- 客户端临时消息在服务器确认后与正式消息合并，不能出现重复气泡。
- 实时方式不可用时可以降级为有上限的增量拉取，并显示连接状态。
- 本地缓存按 NAS 和账号隔离；退出或删除 NAS 配置时清理相应会话秘密，并按用户选择处理可再生媒体缓存。

## 7. 安全与隐私

- Release 构建只允许 HTTPS，沿用岚仓现有证书核对和 QuickConnect 安全边界。
- Chat 会话和必要令牌进入平台安全存储，不得进入偏好设置、普通缓存或日志。
- 加密私钥、会话密钥、解锁口令和恢复材料使用平台安全存储，并按 NAS、账号和设备隔离。
- 密码、OTP、SID、SynoToken、Cookie、消息正文、附件名、群聊名、成员名和真实服务器地址不得进入诊断日志。
- 图片、视频、文件和语音消息下载使用认证请求，不把会话或令牌拼接到 URL。
- 媒体缓存和缩略图按账号隔离，设置大小上限，并允许安全清理。
- 群聊创建、消息发送和附件发送具备权限检查、客户端请求 ID、重复提交保护和结果校验。
- 提醒、投票和加密会话写操作同样具备权限检查、重复提交保护与结果核对。
- 麦克风权限只用于用户主动录制语音消息，不允许后台录音；取消录制后删除临时音频。
- 不提交未脱敏 HAR、PCAP、Chat 响应、真实消息、真实附件和真实用户数据。
- 不复制或分发群晖客户端中的受版权保护界面资源；使用平台系统图标和独立原生实现。

## 8. 实施阶段

### CH0：协议与会话探测

- 使用专用测试 NAS、专用普通账号和虚构消息数据。
- 验证 Chat Server 套件版本、用户列表、用户会话和实际访问端点。
- 验证一对一会话、私人群聊创建、文字发送、图片/视频/文件附件、语音消息、提醒、投票和消息分页。
- 单独验证加密会话的设备密钥、解锁、恢复、轮换、成员变化和多设备互操作。
- 确认同步方式、排序、重复提交、上传限制和错误语义。
- 建立完全脱敏的契约样本、功能开关和兼容矩阵记录。

当前进度：第一批真实调用链已经实现，覆盖能力发现、DSM 会话复用、用户、会话、历史分页、文字发送、群聊创建/邀请和提醒设置；自动化使用完全虚构响应验证解析、正文参数和重复提交保护。下一步由专用测试 NAS 对照实际返回逐项修正，再继续附件、语音、投票和加密协议。

出口：目标能力全部证明可实现；一次性探测代码、HAR、PCAP、临时账号资料和生成物已删除。

### CH1：macOS 一对一文字聊天

- “消息”一级入口、会话列表和聊天界面。
- 用户列表、新建一对一聊天、历史分页、文字收发和草稿。
- 发送中、已发送、失败重试、无权限、离线和会话失效状态。

出口：两个普通账号之间可以稳定收发文字和 Unicode Emoji，不会重复发送。

### CH2：macOS 私人群聊

- 多选用户、填写群聊名称和创建群聊。
- 群聊列表、成员显示、群聊消息历史和多人文字收发。
- 重复创建保护、权限变化和成员不可用的恢复路径。

出口：至少三个专用测试账号完成建群和多人连续收发验证。

### CH3：macOS 图片、视频、文件和语音消息

- 系统选择器、发送前预览、上传进度、取消和失败重试。
- 图片查看器、视频播放器、文件下载、打开和显示所在位置。
- 麦克风权限、录音、试听、发送、播放、取消和临时音频清理。
- 可见范围缩略图、缓存上限、磁盘空间检查和资源释放。

出口：图片、视频、普通文件和语音消息在一对一与群聊中均可收发；取消、空间不足、特殊文件名、录音权限和弱网验证通过。

### CH4：macOS 提醒与投票

- 为消息创建、修改和取消个人提醒，并处理到期提示。
- 创建单选或多选投票、参与投票、展示结果和截止状态。
- 处理无权限、重复提交、成员变化、投票截止和服务器版本不支持。

出口：一对一和群聊中的提醒与投票均按 Chat Server 实际能力工作，超时不会重复创建或重复投票。

### CH5：macOS 加密会话

- 实现加密会话发现、首次解锁、设备密钥安全存储和多设备互操作。
- 验证文字、Emoji、图片、视频、文件、语音消息、提醒和投票在加密会话中的支持边界。
- 完成密钥恢复、轮换、成员变化、设备撤销、错误口令和损坏数据处理。

出口：安全评审和跨设备实机验证通过；任何失败均不回退明文。

### CH6：macOS 稳定性与发布验收

- 长时间同步、网络切换、NAS 切换、前后台和应用退出验证。
- 大量消息和媒体的内存、CPU、滚动、缓存和输入响应验证。
- 浅色/深色、键盘、VoiceOver、动态文字和降低动态效果验收。
- 更新 DSM/Chat Server 兼容矩阵和用户文案。

出口：本计划定义的全部功能达到 macOS 发布门槛。

### CH7：其他平台对齐

- iPhone 和 iPad 复用 Apple Chat Core、Repository、同步与安全实现，使用平台原生触控和自适应布局。
- Android 使用 Kotlin、Jetpack Compose、OkHttp 和 Android Keystore 实现共同契约。
- Windows 使用 C#、WinUI 3、HttpClient 和 Credential Locker/DPAPI 实现共同契约。

## 10. 发布门槛

macOS 首版只有满足以下条件才能标记“已完成”：

- 一对一聊天和私人群聊均可稳定收发文字、图片、视频、文件、语音消息和 Unicode Emoji。
- 消息提醒和投票通过普通账号、弱网、重复提交和结果同步验证。
- 加密会话完成密钥生命周期、安全评审和跨设备验证，失败时不允许明文降级。
- 新建一对一会话不会产生重复会话，群聊创建不会因超时重复提交。
- 发送状态、失败原因和恢复操作对普通用户清楚可见。
- 目标 DSM build 与 Chat Server 版本有完全脱敏的兼容记录。
- 内部接口按套件版本受功能开关保护，Chat 失败不影响文件和照片模块。
- Release 构建没有秘密日志、真实数据、调试代理信任和一次性探测代码。
- 浅色/深色、键盘、触控、VoiceOver/屏幕阅读器、动态文字和降低动态效果验收通过。
